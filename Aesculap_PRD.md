# Aesculap — Hermes Agent 自愈插件 PRD

> Aesculap（蛇杖 / Rod of Asclepius）：为 Hermes Agent 框架设计的自诊断、自修复插件。
> 实时检测系统异常,能自己修的自己修,修不动的升级到外部编码工具,搞不定的喊运维。
> 本文档为开源仓库 PRD,所有示例均为占位,不含任何真实部署信息。

---

## 0. 一句话定位

让 Hermes 实例具备「察觉自己出问题 → 在安全边界内自我修复 → 边界外停手喊人」的能力,默认无人值守运行,只在真正动手或需要人时才打扰运维。

---

## 1. 核心设计原则（实现时不可妥协）

**LLM 提议,代码拍板。**

LLM 只负责两件事:诊断（这是什么问题）和提议（建议怎么修）。「能不能动手、动手到什么程度」由确定性代码强制裁决。LLM 输出 `self_fix` 不等于授权执行;它必须先过一道纯代码的闸门。这是整个插件的安全地基,所有后续逻辑围绕它展开。

理由:LLM 对「我能不能修好这个 bug」的自我评估不可靠（过度自信、无法可靠自评）。因此插件的安全性不能建立在「让 LLM 判断得够准」上,而要建立在「就算 LLM 判断错了,代码也不让它越界」上。

---

## 2. 触发架构（事件驱动 daemon,非定时轮询）

Aesculap 是一个常驻 daemon,**由 systemd 托管**(崩溃自动拉起,解决「监控自己挂了谁管」)。单进程内并行三套探测,共享一条事件队列:

| 探测机制 | 频率 | 抓什么 | 成本 |
|---------|------|--------|------|
| **日志监听**(主力) | 实时 `tail -F` | Traceback / CRITICAL / 配置的 error 模式 | 近零 |
| **存活探测** | 每 1–5 分钟 | 进程被 OOM 等悄悄杀掉（日志可能无痕） | 近零 |
| **全量体检**(兜底) | 默认 24h(可配) | 「不报错的坏」:进程活着但卡死、key 过期未触发调用、磁盘渐满;顺带自检 daemon 健康 | 近零 |

设计要点:
- 三套并入**一个进程**,不要拆成三个 cron。
- 日志监听是主力,解决「报错实时发生但要等下次轮询才发现」的延迟问题。
- 存活探测单列,因为进程被 OOM 杀掉时日志里可能什么都没留,只靠日志抓不到。
- 全量体检从「主力」降级为「兜底」,专抓静默故障。

**前置依赖**:日志监听要求 Hermes 把输出落到日志文件。若 Hermes 当前仅输出到终端(如 screen 内),安装向导需检测并提示先配置 file logging。

---

## 3. 检测层（Tier 0,无 LLM）

一组确定性探针,每个只输出 `OK / WARN / FAIL` + 证据(相关日志行、退出码、指标值)。全 `OK` 则静默,零开销。

内置探针(可扩展):
- Hermes 进程 / session 是否存活
- 心跳文件时间戳是否新鲜(若 Hermes 写心跳)
- 日志近 N 行 error 模式计数
- 模型 API 连通性(轻量 ping 或最后一次成功调用时间戳)
- 通知通道(messaging gateway)是否响应
- 磁盘 / 内存余量

探针定义放配置文件,支持用户增删。探针本身不准调用 LLM。

---

## 4. 去抖动 / 升级判定（唤醒 LLM 之前）

报错事件进入队列后,**不立即唤醒 LLM**。先做廉价确认:

1. 跑一遍相关 Tier 0 探针。
2. 等待 + 重查(默认 60s):这个报错还在吗?Hermes 自己缓过来没?

只有**持续存在**的故障才升级到 Tier 1。瞬时抖动(偶发 503、网络闪断、Hermes 自愈)在这一步被吸收,不烧 token。

> 注:即使采用实时日志监听,模型 provider 偶发抽风仍会进报错流,因此本步骤不可省。阈值(连续次数 / 持续时长)放配置,默认连续 2 次或持续 >60s。

## 5. 分诊层（Tier 1,LLM）

### 5.1 分诊模型

- **安装时由用户选择**。安装向导先自检系统已配置哪些模型 provider,列出来引导用户选,并附建议。
- **建议**:分诊是安全相关的判断环节,且仅在真出故障时触发(低频、成本封顶),不要为省钱用最便宜的小模型,优先用强模型。到 Tier 2 执行自修那步再切廉价模型。
- 插件 provider-agnostic,不硬编码任何特定模型。

### 5.2 分诊输出（结构化,禁止自由发挥）

```json
{
  "diagnosis": "string，问题根因描述",
  "blast_radius": "restart | single_file | multi_file | infra | unknown",
  "reversible": true,
  "confidence": 0.0,
  "route": "self_fix | coding_agent | human | report_only",
  "needs_human_reason": "null | missing_key | needs_payment | needs_human_action | ambiguous | repeated_failure",
  "actions": ["拟执行的具体动作列表"]
}
```

`confidence` 仅供记录与观察,**不作为闸门**。真正的闸门是动作的类别(见 §6)。

## 6. 修复路由与决策闸门

### 6.1 三条路由

| route | 触发条件 | 由谁执行 |
|-------|---------|---------|
| `self_fix` | 已知操作或单处一眼可见的改动,能一步描述完,改完可立即验证 | 分诊模型 / 用户指定模型 |
| `coding_agent` | 需读懂代码库、跨多文件、跑测试、来回迭代 | 外部编码工具(Claude Code / Codex 等,若已安装) |
| `human` | 命中人工介入条件(见 §8) | 通知运维 |
| `report_only` | 无需修复(如瞬时抖动已恢复)或仅需记录 | 仅写日志 |

典型 `self_fix`:进程死了重启(幂等可逆)、配置 JSON 损坏或一眼可见的 typo(单文件改 + 重启)。

### 6.2 代码层闸门（覆盖 LLM 提议）

按**爆炸半径**裁决,不按「难度」:

- LLM 填了 `self_fix`,但 `blast_radius ∈ {multi_file, infra}` 或 `reversible == false` → **强制降级**为 `coding_agent` 或 `human`。
- `blast_radius == unknown` → 不准自修,升级。
- 任何动作命中硬绊线(见 §8.1)→ 直接转 `human`,无视 LLM。

### 6.3 升级阶梯（只升不降）

```
self_fix（最多 3 次，见 §7.2）
  → 仍失败 → coding_agent
    → coding_agent 失败 / 未安装 → human
```

### 6.4 能力清单

启动 / 安装时探测可用工具(`which claude`、`which codex` 等)、检查相关 key 是否存在、检测已配置模型,生成能力清单。若某修复需要 coding_agent 但系统未安装,该路由自动转为 `human`(提示去安装 / 配置)。

## 7. 修复流程

### 7.1 标准流程（动手必备）

```
1. 备份         改配置前先备份原文件；走 git 的改动落 commit
2. 执行修复
3. 全量验证     重跑「整套」探针，不只挂掉那个，防止修 A 弄坏 B
4. 观察窗口     验证通过后留一个观察窗口再宣布成功
5. 失败处理     验证不过 → 先重新诊断为什么不过（可再尝试一次，计入预算）
                → 仍不行 → 回滚到备份 → 升级到下一阶梯
```

**备份 / 回滚是第一版的必选项(非可选)**:

- 配置文件:改前复制原文件到备份区,验证失败原样还原。
- 代码改动:要求在 git 仓库内操作,验证失败 `git revert` / reset。
- **前置依赖**:若 Hermes 代码不在 git 仓库内,安装第一步先 `git init`,所有 coding_agent 改动以 commit 形式落地。

### 7.2 重试预算

- 同一个 bug,`self_fix` 最多尝试 **3 次**。
- 3 次未修复 → 转 `coding_agent`;coding_agent 失败 / 未安装 → 转 `human`。
- 防止无上限重试导致的 token 消耗与「越修越坏」。

### 7.3 级联保护

若一次有多个探针同时 `FAIL`(典型如即将 OOM),判定为系统级故障,**不逐个修**,直接升级 `human`。

## 8. 人工介入（喊人）

### 8.1 硬绊线（纯代码,与 LLM 无关）

任何动作命中以下,直接拦下转 `human`,LLM 越权也跨不过:

- 写入 `chmod 444`(只读)文件
- 触碰 `.env` / key / 凭证文件
- 执行 `rm`、`git push --force`
- 命中 §9 可写范围之外的路径
- 命中付费 / 计费接口
- 修改 Aesculap 自身目录(见 §9.3)
- 修改身份文件(见 §9.2)

### 8.2 LLM 识别的信号

分诊 prompt 教模型识别并 emit `route: human`:

- 日志出现 `401 / invalid api key / unauthorized` → key 缺失或过期
- 需点验证链接 / CAPTCHA / OAuth / 2FA → 仅人可完成
- 需付费(续费、充值余额、购买资源)
- 意图模糊,需产品 / 业务判断
- 同一 bug 反复失败(`repeated_failure`)

### 8.3 通知内容（小白友好,可操作）

喊人通知必须包含:

1. **哪儿坏了** — 故障描述 + 触发探针
2. **试了什么** — 已尝试的修复动作及结果
3. **要你做什么** — 具体到可操作的路径:哪个 key、哪个文件的哪一行、哪个链接、要点哪个按钮
4. **修复指引** — 尽量给一行「执行 X 即可」级别的指引,面向非技术用户

**Key 安全规则(强制)**:涉及 API key / 凭证时,只告诉用户 key 填到哪个文件 / 哪个位置,给出格式示例,**绝不要求用户把 key 贴进聊天窗口或日志**。

通知通道走 Hermes 现有的 messaging gateway。

## 9. 可写范围与黑名单

### 9.1 安装时强制选档（开源关键设计）

安装向导**不提供默认值,强制用户显式选择**一档可写范围,并展示风险星级与警告:

| 档位 | 可写范围 | 风险 | 适用 |
|------|---------|------|------|
| A | 仅项目树内 | ⭐ | 最安全,推荐默认选择 |
| B | Hermes 配置目录及项目树 | ⭐⭐⭐ | Hermes 专用主机 |
| C | 整个环境 / 主机 | ⭐⭐⭐⭐⭐ ⚠️ | 仅当整机专供 Hermes 时 |

**⚠️ 强警告**:若插件安装在个人电脑或与其他重要数据共存的环境,**严禁选 C 档**。向导检测到非专用环境迹象时应额外二次确认。

### 9.2 黑名单（无视档位,永远生效,default-allow）

选档决定「可写范围的外边界」;边界内采用 default-allow(除黑名单外都可改)。黑名单硬写死:

- 身份文件(如 `soul.md` / 人格 / 记忆架构相关文件)
- `.env` / key / 凭证文件
- `chmod 444` 只读文件
- Aesculap 自身目录(见 §9.3)
- 系统级敏感路径(C 档下仍排除系统关键目录)

> 注:cron / 日常自修基本不涉及身份文件,身份文件入黑名单写死,只读不可改,需改时一律喊人。

### 9.3 禁止自我修改

Aesculap 不得修改自身任何文件(递归自修 = 灾难)。自身目录硬入黑名单。Aesculap 自身崩溃由 systemd 负责拉起。

## 10. 配置与开关

### 10.1 主开关

用户可一键启停整个插件(配置项 / flag 文件)。

### 10.2 模式位

- `fix`(默认):检测 + 自修 + 必要时喊人。这是插件存在的意义。
- `observe`(安全档):只检测 + 报告,绝不动手。仅作为「发现它判断飘了」时的降级档,非常态。实现成本低,保留。

### 10.3 主要可配项

触发阈值(去抖动连续次数 / 持续时长)、全量体检周期、存活探测周期、重试预算次数、分诊模型、自修模型、coding_agent 选择、可写范围档、探针定义、通知通道。

---

## 11. 通知策略

- 仅在**真正动手修复**和**需要喊人**时推送。
- 健康的静默 run 不打扰用户(防过度打扰)。
- 不做每日摘要 / 日报。

通知通道失效兜底:若判定通知通道(gateway)本身故障,优先抢修通道(Hermes 自身已具备 gateway 自重启能力,Aesculap 复用即可)。整机全挂超出 Aesculap 能力范围,由 Hermes 现有通知机制 + 运维兜底。

---

## 12. 安全护栏汇总

| 护栏 | 作用 |
|------|------|
| 动手前备份 | 修错可还原 |
| 全量验证 + 观察窗口 | 防止修 A 坏 B、防止假性修复 |
| 失败先诊断后回滚 | 不盲目回滚,先理解为何失败 |
| 重试预算(3 次) | 防 token 篝火、防越修越坏 |
| 并发锁(lockfile) | 防 daemon 在上次 coding_agent 未完成时重复触发(其运行可达数分钟) |
| 去重 + 冷却 | 维护 open issues 状态文件;已报告待处理的不重复刷屏 |
| 级联保护 | 多探针同挂 → 判系统级 → 喊人 |
| 硬绊线 | LLM 越权也跨不过的代码级红线 |
| systemd 托管 | daemon 自崩自起 |

## 13. 审计日志

- append-only,记录全链路:检测事件 → 分诊决策 → 执行动作 → 结果。
- 用户可随时审阅插件做过什么。
- 日志备份同步到备份节点 / 对象存储(可配)。

---

## 14. 开源注意事项

- **provider-agnostic**:不硬编码任何模型、不硬编码任何通知通道,全部走配置 + 安装时检测。
- **跨环境安全**:§9.1 强制选档是核心,默认偏保守(推荐 A 档)。
- **脱敏**:仓库内不含任何真实 IP / 域名 / 凭证 / 部署信息,示例用占位符。
- **README**:面向开源用户说明三档风险、强警告(个人电脑勿选 C)、安装前置(file logging、git init)。

## 15. 第一版范围（MVP）

按用户拍板,第一版**直接实现自诊断 + 自修**,不走影子模式渐进上线,但安全件齐备:

包含:
- 事件驱动 daemon(三套探测)+ systemd
- 去抖动确认
- Tier 0 探针 + Tier 1 分诊(结构化输出)
- 修复路由 + 代码闸门 + 升级阶梯
- 备份 / 验证 / 回滚 + 3 次重试预算
- 安装向导强制选档 + 黑名单 + 禁止自改
- 硬绊线 + 喊人通知(可操作 + key 安全)
- 主开关 + observe 安全档
- 并发锁 + 去重冷却 + 级联保护
- 审计日志

可放二期:
- 探针 / 修复模式的社区可扩展插件机制
- 更细的成本统计与可选摘要
- coding_agent 的多工具自动选择策略优化

## 16. 给实现方（CC）的指引

1. 先确认两项前置:Hermes 是否已 file logging;Hermes 代码是否在 git 仓库。缺则向导补齐。
2. 自主拆分实现阶段,**分阶段与架构决策者确认**后再推进,不要一口气全写完。
3. §1（LLM 提议代码拍板）、§6.2（代码闸门）、§7.1（备份回滚）、§8.1（硬绊线）、§9（可写范围与黑名单）为安全核心,实现时优先且不可简化。
4. 待澄清项在实现前列出,逐条确认,不自行假设。
